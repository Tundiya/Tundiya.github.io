SQL querying and managing data

OKAY!!!!!!Just finished SQL module. Interesting stuff. I am wondering how many back ends use SQL, once again the language itself seems very intuative with creation of the tables and the querying of said tables. Joining tables together, with outer and inner joins, left side and right side joins, this all starts to really make senes after having some practice with the commands and getting comfortable with the idea that you are literally writing into tables and organizing data in ever more interesting or useful ways! I started off kind of struggling with this perhaps because it was my jump into a third langauge after only programming for a few months seriously. However, like I mentioned above I was given the chance to use it over and over and finally towards the end of the module it very much began to flow out of me much more easily. Once again not giving up and repitition prevails! I have also began to try a new avenue of memorization of programming concepts and commands. For the last two years, while I have been at work and as a way to help better develope my memorization skills, after I finish listening to a podcast I try to do a review of it in my own head immediatly after ends. In particular I go over who was in it, what were the main themes, what was new information I learned, what was information that I may have changed my mind about, and things of that nature. I repeat this process for every podcast I listen to, multiple times a day. This REALLY has been helpful for me! So I have decided to apply a similar technique with programming. As on my three work days I really don't get a serious chance to sit down and do some studying I try to help myself as much as possible while at work. So in my head, when safe to do so, I review the concepts and commands and their applications repeatedly throughout the day to help better make new synaptic connections in my brain. In other words commit this stuff to memory! I'm sure this will help as the general approach has already been amazing. BUT ANYWAY back to SQL...fundamentally the idea very much makes sense to me. Make tables to drive data into them, try to construct the tables so the data can be comfortably parsed together if need be. Query the tables for desired information, INSERT into them or DELETE, only when appropriate. Pushing new table rows or columns into an already existing table is preferred to updating the initial table build so the existing data does not get deleted or corrupted. There is an importance to strictly giving out permissions to read/write into the tables as they may contain sensitive personal info or damage can be done the the data. Partial permissions can be given, in general however, access should be limited to back-end engineers. I honestly thought this would be super boring and maybe at first it was but concept is certainly powerful. I'm not sure where in software, if at all, I will end up but I want to try and not forget this stuff. If at all practical perhaps since this blog is in itself a way to keep myeslf at least somewhat proficient in HTML/CSS, I wonder if I can run some SQL table along side this blog...perhaps to push basic data into it from even my own usage or something like that. Just some sort of reference I can occasionaly get into. Perhaps, trying to keep up with every language I learn along the way is impractical, it may be. Too soon to tell for sure let us see where this endeavor goes. Also, from here on out I have a new layout built for the blog using GRID structure instead of arbitrary YOLO style.
